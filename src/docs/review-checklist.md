# 全项目审查检查清单

## 一、代码简洁性

- [ ] 是否存在可以合并的重复逻辑（DRY 违反）
- [ ] 是否存在只调用了一行的包装函数（无意义抽象）
- [ ] 是否存在只有一个实现的接口/类型（过度抽象）
- [ ] 是否存在可以用更短写法替代的代码（三元替代 if-else、可选链替代 && 等）
- [ ] 是否存在未使用的变量、函数、类型、import
- [ ] 是否存在可以内联的单行组件或函数
- [ ] 是否存在硬编码值应提取为常量但没提取的
- [ ] 是否存在已提取为常量但只使用了一次的（过度提取）

## 二、实现方式

- [ ] 是否有更好/更原生的实现方式（CSS 替代 JS 动画、原生 API 替代手动实现等）
- [ ] 是否存在可以用现有依赖解决却手动实现的逻辑
- [ ] useEffect 是否可以替换为派生状态（useMemo / 直接计算）
- [ ] useState 是否可以替换为 useRef（不需要驱动渲染的值）
- [ ] 是否存在不必要的客户端状态（可由 URL / props / context 直接获取）
- [ ] 是否存在应该用 CSS 处理却用 JS 处理的交互（hover、focus、动画）
- [ ] 是否存在 props drilling 过深，应考虑 context 或组合模式

## 三、风格与主题

- [ ] 颜色是否全部使用 CSS 变量（`var(--xxx)`），无硬编码色值
- [ ] 间距是否遵循一致的节奏（4/8/12/16/24/32 等）
- [ ] 圆角是否统一使用 `var(--radius)` 或一致的值
- [ ] 字体大小是否遵循统一的阶梯（xs/sm/base/lg/xl 等）
- [ ] 过渡动画时长是否一致（0.15s / 0.2s / 0.3s）
- [ ] 组件间视觉风格是否统一（卡片、按钮、输入框等）
- [ ] 图标风格是否统一（线宽、大小、配色）

## 四、潜在问题

- [ ] 竞态条件（快速切换/连续操作导致数据错乱）
- [ ] 内存泄漏（未清理的 timer、listener、subscription）
- [ ] 闭包陷阱（useEffect/useCallback 引用过期状态）
- [ ] 未处理的边界情况（空数据、超长数据、特殊字符）
- [ ] XSS 风险（dangerouslySetInnerHTML、未转义用户输入）
- [ ] 未做输入校验的 API 端点
- [ ] 错误处理是否完整（网络失败、服务端异常、用户误操作）
- [ ] 异步操作缺少 loading 状态或重复提交保护

## 五、遗留问题

- [ ] 是否有 TODO / FIXME / HACK 注释未解决
- [ ] 是否有注释掉的代码未删除
- [ ] 是否有废弃但未删除的文件（无任何引用）
- [ ] 是否有废弃但未删除的 CSS 类（无任何使用）
- [ ] 是否有旧版本的兼容代码已不再需要
- [ ] 是否有临时方案（workaround）未替换为正式方案

## 六、代码结构

- [ ] 文件职责是否单一清晰
- [ ] 目录组织是否合理（按功能分组 vs 按类型分组）
- [ ] 组件是否过大应拆分（>200 行考虑拆分）
- [ ] 是否存在循环依赖
- [ ] 公共组件与页面私有组件是否分清
- [ ] 类型定义是否放在合理位置（就近 vs 集中）

## 七、代码注释

- [ ] 注释是否解释"为什么"而非"是什么"
- [ ] 是否存在显而易见无需注释的代码被注释了
- [ ] 是否存在复杂逻辑缺少注释
- [ ] 注释是否与代码保持同步（代码改了注释没改）
- [ ] 中文注释是否简洁清晰，无废话

## 八、可优化推荐

- [ ] 是否可以添加骨架屏 / 加载状态提升体验
- [ ] 是否可以添加过渡动画提升体验
- [ ] 是否可以添加键盘快捷键
- [ ] 是否可以添加空状态提示（无数据时）
- [ ] 是否可以优化 SEO（meta、OG、结构化数据）
- [ ] 是否可以优化无障碍（aria、语义标签、focus 管理）
- [ ] 是否可以优化性能（懒加载、虚拟滚动、缓存策略）
- [ ] 是否可以优化移动端适配
