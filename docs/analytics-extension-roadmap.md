# 网站统计功能扩展方向

> 生成日期：2026-06-09
> 基于当前知简项目分析系统的完整调研，对照 Plausible / Umami / Matomo / GoatCounter 等主流工具

---

## 一、已收集但未展示的数据

系统**已经在采集**这些数据，但仪表盘没有用到：

| 数据 | 存储位置 | 潜在用途 |
|------|----------|----------|
| **页面标题** (`title`) | events 表 | 访问记录里显示文章标题而非裸路径，更易读 |
| **浏览器语言** (`lang`) | events 表 | 语言分布图，了解读者语言偏好 |
| **屏幕分辨率**（原始值如 `1920x1080`） | events 表 | 目前只做了粗分 Mobile/Tablet/Desktop，可展示分辨率分布 |
| **新访客标记** (`is_new`) | events 表 | 新访客 vs 回访者比例，衡量站点粘性 |
| **会话 ID** (`session_id`) | events 表 | 每次访问的页面深度、会话时长、入口/出口页面分析 |
| **每日每页的跳出率 & 平均停留** | daily 表 | 热门页面排行中已有聚合数据但未返回，可展示每页的跳出和停留 |
| **离开事件** (`type='leave'`) | events 表 | 出口页面分析——用户最后看了哪页离开 |

## 二、主流工具都有的核心特性（你目前没有）

| 功能 | Plausible | Umami | Matomo | 实现难度 | 价值 |
|------|-----------|-------|--------|----------|------|
| **转化目标（Goals）** | ✅ 代码+无代码 | ✅ | ✅ | 中 | ⭐⭐⭐⭐⭐ 核心功能——知道多少人完成了注册、购买、订阅等关键动作 |
| **转化漏斗（Funnel）** | ✅ | ✅ | ✅ 高级 | 中 | ⭐⭐⭐⭐⭐ 用户从首页→文章列表→文章→评论，哪一步流失最多？ |
| **用户路径/旅程** | ✅ | ✅ | ✅ User Flow | 高 | ⭐⭐⭐⭐ 看到用户实际走的路线，而非假设的路线 |
| **浏览器 & OS 分布** | ✅ | ✅ | ✅ 30+报告 | 低 | ⭐⭐⭐⭐ 前端兼容性决策必备 |
| **地理位置（国家/城市）** | ✅ | ✅ | ✅ 世界地图 | 中 | ⭐⭐⭐⭐ 了解读者从哪来 |
| **实时在线人数** | ✅ 每30秒刷新 | ✅ | ✅ Live | 中 | ⭐⭐⭐ 很直观，但心跳机制需要改架构 |
| **出口页面分析** | ✅ | ✅ | ✅ | 低 | ⭐⭐⭐ 用户最后看了哪页离开？ |
| **入口页面分析** | ✅ | ✅ | ✅ | 低 | ⭐⭐⭐ 用户从哪进入站点？ |
| **新访客 vs 回访者比例** | ✅ | ✅ | ✅ | 低 | ⭐⭐⭐ 衡量站点粘性，数据你已有（is_new） |
| **邮件/Slack 定期报告** | ✅ | ❌ | ✅ PDF+HTML | 中 | ⭐⭐⭐ 自动推送周报/月报 |
| **共享/嵌入仪表盘** | ✅ | ✅ | ✅ 嵌入widget | 中 | ⭐⭐ 对外展示流量数据 |

## 三、已有数据但没展示/没充分利用的

| 功能 | 当前状态 | 改动量 | 价值 |
|------|----------|--------|------|
| **页面标题显示** | 存了 title 但访问记录只显示 path | 极低 | ⭐⭐⭐⭐⭐ `/blog/tea-ceremony` 不如「茶道之美」直观 |
| **每页跳出率 & 平均停留** | daily 表已聚合，getPageRank 未返回 | 低 | ⭐⭐⭐⭐ 知道哪篇内容留住人 |
| **语言分布** | 存了 lang 但没查询 | 低 | ⭐⭐⭐ 了解国际读者比例 |
| **屏幕分辨率分布** | 只做了 Mobile/Tablet/Desktop 三类 | 低 | ⭐⭐⭐ 精细布局适配 |
| **会话深度（页面数/会话）** | 存了 session_id 但没分析 | 中 | ⭐⭐⭐⭐ 平均每次访问看几页？ |
| **会话时长** | 存了 duration + session_id | 中 | ⭐⭐⭐⭐ 比单页停留更有意义 |

## 四、高级特性（Matomo 级别，适合未来迭代）

| 功能 | 说明 | 实现难度 | 价值 |
|------|------|----------|------|
| **热力图（Heatmap）** | 在页面上叠加点击/滚动热区图 | 高（需前端截图+坐标追踪） | ⭐⭐⭐⭐⭐ UX 优化核心工具 |
| **会话录制（Session Recording）** | 录制并回放用户操作过程 | 极高（需 DOM 变更追踪+回放引擎） | ⭐⭐⭐⭐⭐ debugging UX 问题的终极武器 |
| **A/B 测试** | 对比两个版本的效果差异 | 高 | ⭐⭐⭐⭐ 需要先有 Goals/Funnel |
| **表单分析** | 哪个字段导致用户放弃填写 | 高 | ⭐⭐⭐⭐ 需要表单交互追踪 |
| **留存分析（Cohort）** | 某周来的用户，后续几周还回来吗 | 中 | ⭐⭐⭐⭐⭐ 衡量长期价值的关键指标 |
| **自定义维度** | 给用户/动作打标签（会员/游客/付费） | 中 | ⭐⭐⭐⭐ 细分群体 |
| **站内搜索分析** | 用户搜了什么关键词 | 中 | ⭐⭐⭐⭐ 内容规划依据 |
| **页面过渡** | 进入某页之前和之后去了哪 | 中 | ⭐⭐⭐ |
| **注释** | 在时间线上标注事件（如「上线新功能」） | 低 | ⭐⭐⭐ 帮助理解数据变化原因 |
| **自定义报告** | 用户自选维度+指标组合 | 高 | ⭐⭐⭐ |
| **预警通知** | PV 突然暴涨/暴跌时自动告警 | 中 | ⭐⭐⭐⭐ 运维必备 |

## 五、轻量隐私友好型工具的特色（Plausible/Umami 级别）

知简的系统和 Plausible/Umami 定位最接近——轻量、隐私优先、自托管。这些是最值得参考的：

| 功能 | 为什么适合你 |
|------|-------------|
| **自动追踪外链点击** | Plausible 无需代码自动追踪，script.js 也可以拦截 `click` 事件判断是否外链 |
| **自动追踪文件下载** | 同上，拦截 PDF/ZIP 等下载链接 |
| **滚动深度追踪** | Plausible 自动做 1-100% 滚动追踪，可以加到 script.js 中 |
| **AI 流量监控** | Plausible 专门追踪 ChatGPT/Perplexity/Claude 来源——这是 2025-2026 的新趋势 |
| **UTM 参数解析** | 自动归类为付费搜索/社交媒体/直接访问等渠道分组 |
| **SPA 导航追踪** | 已有 pushState/popstate 监听 |
| **无代码目标** | Plausible 可以把「访问 /thank-you 页」直接定义为目标，无需写 JS |
| **轻量脚本** | Plausible 脚本 <7KB，script.js 也可以保持极小体积 |

## 六、2025-2026 新趋势特性

| 功能 | 说明 |
|------|------|
| **AI 流量来源** | 来自 ChatGPT、Perplexity、Claude 等 AI 工具的引用流量，Plausible 已专门做这个分类 |
| **Core Web Vitals** | LCP/FID/CLS 性能指标追踪，Google 排名影响因素 |

| **服务器端追踪** | 不依赖客户端 JS，服务端直接处理——绕过浏览器隐私限制和广告拦截器 |
| **MCP 集成** | Matomo 已经支持 MCP 协议——让 AI 助手直接查询统计数据 |

## 七、可新增采集的数据

| 类别 | 数据 | 采集方式 | 价值 |
|------|------|----------|------|
| **浏览器 & OS** | Chrome 120 / macOS / Windows | `navigator.userAgent` | 浏览器分布，前端兼容性决策 |
| **地理位置** | 国家 / 省份 / 城市 | 服务端 IP 解析（GeoIP） | 读者地域分布 |
| **视口尺寸** | 浏览器实际可视区域 | `clientWidth x clientHeight` | 比屏幕分辨率更准确地反映布局适配 |
| **设备像素比** | Retina (2x) / 标准 (1x) | `devicePixelRatio` | 图片/图标适配决策 |
| **网络类型** | 4G / WiFi / 3G | `navigator.connection` | 性能优化参考 |
| **UTM 参数** | utm_source / utm_medium / utm_campaign | 解析 URL query | 营销活动追踪，衡量推广渠道效果 |
| **页面加载性能** | TTFB / FCP / LCP | `PerformanceObserver` | 技术健康指标 |
| **滚动深度** | 读到 25% / 50% / 75% / 100% | Scroll 事件 | 内容消费深度，区分"真阅读"和"假跳出" |
| **外链点击** | 点击了哪些出站链接 | 事件拦截 | 了解用户流向 |
| **文件下载** | PDF/ZIP 下载次数 | 事件拦截 | 资源使用统计 |
| **自定义事件** | 按钮点击、表单提交等 | `window.zhijian.track()` API | 转化漏斗、用户行为分析 |
| **实时在线** | 当前正在浏览的人数 | 启用心跳机制 | 实时仪表盘 |

## 八、架构层面待改进项

1. **心跳机制缺失** — schema 里定义了 `heartbeat` 事件但客户端从未发送。长时间停留的页面（用户读了 30 分钟）如果 `leave` 事件没触发，时长就会丢失
2. **来源/设备的实时查询** — 目前来源排行和设备分布每次都从 events 原始表查，数据量大后会变慢，应该聚合到 daily 表
3. **无数据清理机制** — SQL 注释说 90 天保留，但没有自动清理任务
4. **客户端时间戳被丢弃** — `ts` 字段收到但没存，所有时间用服务端时钟，跨时区用户可能出现日期边界问题
5. **每页跳出率/停留已有但未返回** — `getPageRank()` 只返回 PV/UV，daily 表里已经聚合了每页的 bounce 和 avg_duration

## 九、推荐实施路线

### 阶段 1（低成本高回报，1-2 天）

- 访问记录显示页面标题 + 新/回访标记
- 热门页面增加每页跳出率和平均停留
- 新增浏览器 & OS 采集 + 展示
- 入口/出口页面分析
- 语言分布图

### 阶段 2（中等投入，3-5 天）

- UTM 参数解析 + 渠道分组
- 转化目标（Goals）+ 无代码目标
- 转化漏斗（Funnel）
- 会话深度 & 会话时长统计
- 新访客 vs 回访者比例
- AI 流量来源分类
- 地理位置解析

### 阶段 3（高投入，分步迭代）

- 滚动深度追踪
- 外链点击 + 文件下载追踪
- 自定义事件 API
- 留存分析（Cohort）
- 实时在线人数
- 邮件定期报告
- 预警通知
