# 博客公开页面审查与处理方案

## 文档目标

本文档用于整理当前对外开放博客页面的审查结果，范围包括：

- 首页 `/`
- 博客列表页 `/blog`
- 博客详情页 `/blog/[slug]`

重点关注以下几类问题：

- 是否存在不合理实现
- 是否有遗留未删除代码
- 是否有可以简化的实现
- 是否有更合适、更稳定的替代方案
- 是否存在潜在 Bug 或线上风险

---

## 当前结论

当前博客公开页面整体已经具备基础的内容展示和 SEO 能力，但还有几处会直接影响：

- 移动端可用性
- 列表页 SEO 一致性
- 详情页请求效率
- 时间字段稳定性
- 页面切换性能体验

建议按优先级分为两批处理：

### 第一优先级

- 修复移动端公共导航不可用问题
- 调整 `/blog` 列表页为服务端主导渲染

### 第二优先级

- 合并详情页重复查询
- 统一日期解析与时区处理
- 收敛公开页链接预取策略

---

## 问题 1：移动端公共导航不可用

### 问题描述

当前公开站点顶部导航在桌面端可用，但移动端只保留了一个菜单按钮，没有任何展开逻辑，也没有对应的移动菜单内容。

这意味着：

- 手机访问首页时，无法正常进入博客列表页
- 手机访问详情页时，无法依赖站点导航回到其他公开页面
- 页面上存在“可点击但无行为”的遗留交互

### 影响范围

- 首页 `/`
- 博客列表页 `/blog`
- 博客详情页 `/blog/[slug]`
- 所有使用 `PublicChrome` 的公开页面

### 相关文件

- `src/components/site/public-chrome.tsx`
- `src/components/site/public-chrome.module.css`

### 风险等级

高

### 建议处理方式

方案 A，补完整移动端导航：

- 为移动端菜单按钮增加打开/关闭状态
- 增加移动端抽屉或下拉菜单
- 复用现有 `PUBLIC_NAV_ITEMS`
- 支持点击空白关闭、点击链接后自动关闭

方案 B，如果暂时不做移动导航：

- 先移除无行为按钮
- 保留最少可用跳转，例如直接展示首页和文章入口

### 推荐落地方式

优先采用方案 A。

这是对外页面的基础可用性问题，不建议继续保留“假按钮”。

## 问题 2：`/blog` 列表页当前更偏客户端交互页，不适合作为完整 SEO 页

### 问题描述

当前 `/blog` 页面虽然在服务端取了文章数据，但分页、分类筛选、标签筛选的核心逻辑都在客户端组件中完成，并依赖 `useSearchParams()`。

同时页面又定义了：

- 服务端 `generateMetadata`
- 列表页 JSON-LD
- `page` 参数的 canonical / prev / next

这会导致页面出现一个问题：

同一个 URL 的 SEO 描述是服务端生成的，但页面实际列表内容是客户端根据参数二次计算出来的，二者不完全同步。

### 典型风险

- 搜索引擎抓取到的 HTML 与最终展示内容不完全一致
- `?page=2` 的 SEO 声明和真实首屏内容可能不一致
- `category`、`tags` 筛选页如果被索引，容易产生重复内容和索引噪音
- `Suspense fallback` 对公开列表页的 SEO 价值不高

### 影响范围

- 博客列表页 `/blog`

### 相关文件

- `src/app/blog/page.tsx`
- `src/app/blog/_components/blog-list-client.tsx`

### 风险等级

高

### 建议处理方式

建议把 `/blog` 分成两层能力：

#### 第一层：真正可索引的分页页

由服务端直接根据 `searchParams.page` 计算：

- 当前页数据
- 总页数
- canonical
- prev / next
- 对应 JSON-LD

这样 `/blog?page=2` 才是真正意义上的可索引分页页。

#### 第二层：筛选能力

分类和标签有两个可选方向：

方案 A，继续保留筛选，但明确这些筛选结果页不作为重点索引页：

- 对筛选态设置更谨慎的 canonical
- 必要时对筛选态做 `noindex`

方案 B，把分类/标签也服务端化：

- 让 `category`、`tags` 进入服务端参数处理
- 按查询参数返回对应 HTML
- 再决定是否允许索引

### 推荐落地方式

先做最小稳定版本：

1. 把分页改为服务端渲染
2. 分类/标签先保留交互，但不作为 SEO 核心页
3. 后续再决定是否做服务端筛选页

### 可简化点

当前 `Suspense` 对这个页面帮助有限，可以在服务端分页改造后一起简化掉。

## 问题 3：博客详情页存在重复查询

### 问题描述

详情页当前在一次请求内会重复获取同一篇文章：

- `generateMetadata` 查询一次
- 页面组件再次查询一次

而 `getPostBySlug()` 内部还会继续补一次标签信息。

这样一个详情页请求实际会产生多次数据库访问。

### 影响

- 响应时间增加
- 数据库压力增大
- 对搜索引擎抓取和高频访问页面不够友好

### 影响范围

- 博客详情页 `/blog/[slug]`

### 相关文件

- `src/app/blog/[slug]/page.tsx`
- `src/lib/posts.ts`

### 风险等级

中高

### 建议处理方式

推荐统一成“单次读取，多处复用”：

方案 A，使用缓存包装读取函数：

- 将按 `slug` 读取文章的逻辑收敛为一个共享方法
- 页面主体和 metadata 共用这一份结果

方案 B，提前在更高层读取：

- 先读取文章
- 再把文章对象分别用于 metadata 和页面主体

### 推荐落地方式

优先采用方案 A。

这样结构更清晰，后续如果文章详情页继续扩展相关推荐、上一篇下一篇，也更容易统一数据入口。

### 可优化点

标签查询也可以继续优化：

- 详情页只查单篇文章时，不一定需要走通用批量补全逻辑
- 可以考虑直接在主查询阶段就把分类和标签展示信息整理好

## 问题 4：日期解析方式存在时区漂移风险

### 问题描述

当前多个地方都把数据库返回的字符串时间直接转成：

- `YYYY-MM-DDTHH:mm:ss`
- 再交给 `new Date()`

这种写法依赖运行环境时区解释，线上如果出现以下任一情况，就可能导致时间偏移：

- MySQL 时区和 Node 时区不一致
- 本地和服务器时区不一致
- 容器环境与系统环境不一致

### 可能表现

- 页面展示日期和预期不一致
- sitemap 中 `lastModified` 不稳定
- 详情页 Open Graph 的 `publishedTime`、`modifiedTime` 可能存在偏差

### 影响范围

- 首页文章日期
- 列表页文章日期
- 详情页发布时间
- sitemap

### 相关文件

- `src/lib/post-shared.ts`
- `src/app/sitemap.ts`
- `src/app/blog/[slug]/page.tsx`

### 风险等级

中

### 建议处理方式

统一一个明确规则：

方案 A，数据库时间统一视为北京时间字符串处理

- 不直接让 `new Date('YYYY-MM-DDTHH:mm:ss')` 自由解析
- 在服务端显式按约定时区处理

方案 B，数据库统一存 UTC，应用层统一转展示时区

- 这是长期更标准的做法
- 但如果当前库里已经大量是本地时间，改造成本更高

### 推荐落地方式

如果项目当前主要面向中文站点，优先采用方案 A，先保证：

- 页面显示稳定
- sitemap 稳定
- 结构化数据时间稳定

后续有国际化或多时区需求时，再升级到方案 B。

## 问题 5：公开站点大量关闭 `Link` 预取，策略分散且收益不明显

### 问题描述

当前公开页中多处链接都显式设置了 `prefetch={false}``：

- 顶部导航
- 博客列表项
- 文本链接组件

这会让站内跳转更依赖实时请求，影响从：

- 首页到列表页
- 列表页到详情页
- 详情页返回其他页面

的切换体验。

### 影响范围

- 所有公开页内部链接

### 相关文件

- `src/components/ui/text-link.tsx`
- `src/components/site/public-chrome.tsx`
- `src/app/blog/_components/blog-list-client.tsx`

### 风险等级

中

### 建议处理方式

建议回归默认策略：

- 公共组件不要默认关闭预取
- 只有极少数明确不适合预取的页面再单独禁用

### 推荐落地方式

先从两个层面收敛：

1. 去掉 `TextLink` 里的默认 `prefetch={false}`
2. 评估公共导航和博客列表是否真的需要关闭预取

如果没有明确证据证明预取带来明显负担，就优先使用 Next 默认行为。

## 可顺手处理的简化项

这些不是最高优先级，但适合在优化时顺手一起收敛。

### 1. 首页和列表页中的图片渲染方式可以统一

当前公开页仍混用了原生 `<img>` 和 Next 的图片组件。

建议：

- 明确哪些图片走 `next/image`
- 哪些图片保留原生 `<img>`
- 在公开博客链路中尽量统一策略

### 2. 列表页参数职责可以进一步收口

当前 `page`、`category`、`tags` 混在客户端 URL 逻辑里。

建议：

- `page` 交给服务端
- `category`、`tags` 明确为“交互筛选”还是“可索引筛选”

### 3. 详情页结构化数据可以继续增强

当前已经有基础 `BlogPosting` 和面包屑。

后续可补：

- 更准确的 `publisher`
- 封面图绝对地址统一生成
- 阅读时长或作者页链接

---

## 推荐执行顺序

建议按下面顺序推进：

1. 修复移动端公共导航
2. 重构 `/blog` 为服务端分页主导
3. 明确筛选页是否索引，并调整 canonical / noindex 策略
4. 合并详情页重复查询
5. 统一时间解析逻辑
6. 收敛 `prefetch={false}` 使用范围
7. 顺手清理可简化项

---

## 建议验收方式

每一项处理后，建议至少做以下验证：

### 导航相关

- 手机宽度下首页、列表页、详情页都能正常跳转

### 列表页相关

- 直接访问 `/blog?page=2` 能返回正确 HTML 内容
- 查看页面源码时，标题、描述、canonical、列表内容一致

### 详情页相关

- 单篇文章访问时数据库查询次数下降
- metadata 与页面主体使用同一份文章数据

### 时间相关

- 首页、列表页、详情页、sitemap 日期一致
- 本地与服务器显示结果一致

### 性能相关

- 首页进入列表、列表进入详情的体感比当前更顺滑

---

## 最终建议

如果只做一轮最值钱的优化，建议直接做这三件事：

1. 修复移动端导航
2. 把 `/blog` 分页改成服务端渲染
3. 合并详情页重复查询

这三项改完之后，公开博客页面在：

- 可用性
- SEO 稳定性
- 性能
- 可维护性

上都会有比较明显的提升。
