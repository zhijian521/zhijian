## 3. 推荐阅读顺序

1. 根目录 `README.md`：了解项目定位、运行方式和常用命令。
2. 《需求文档》：确认当前产品范围和验收边界。
3. 《功能文档》：了解页面入口、权限和用户能力。
4. 对应功能模块文档：查看具体交互和数据规则。
5. 《架构设计文档》：了解分层、依赖和核心数据流。
6. 《开发规范导航》：开始开发前读取相关专项规范。

## 4. 命名与编号规则

- 顶层分类目录使用两位数字编号，表达稳定阅读顺序。
- 分类内部独立编号；`00` 只用于导航索引。
- 文档文件名使用中文，保留 API、Git 等必要技术名称。
- 文件名不包含日期或版本号，核对日期和适用范围放在文档头部。
- 新增文档先确定分类，再使用该分类下一个编号。
- 不因插入单个文件批量重排既有编号，避免无意义链接变化。

## 5. 文档头部格式

正式文档在一级标题后维护：

```markdown
- 文档状态：当前实现基线 / 当前功能说明 / 开发规范 / 设计方案 / 已废弃
- 适用范围：当前项目
- 最后核对：YYYY-MM-DD
- 维护范围：本文负责描述的内容
- 关联代码：相关目录、配置或核心文件
```

只有重新核对正文与当前代码后，才能更新“最后核对”日期。

## 6. 文档拆分规则

- 少于 300 行且主题单一时，优先保留一个文件。
- 超过 400～500 行时，检查是否存在可独立维护的主题，再决定拆分。
- 包含三个以上相互独立且更新频率不同的主题时，可以提前拆分。
- 拆分后的目录保留导航文档，子文档按阅读顺序编号。
- 不为尚未出现的复杂度创建空文件或空目录。

## 7. 同步维护规则

代码和文档必须在同一次提交中保持一致：

| 代码变化                                 | 必须检查或更新                                          |
| ---------------------------------------- | ------------------------------------------------------- |
| 新增、删除或改变用户功能                 | 需求文档、功能文档、对应功能模块文档                    |
| 修改按钮、快捷键、默认值、权限或交互     | 功能文档、对应功能模块文档；影响验收边界时更新需求文档  |
| 修改目录、模块职责、数据模型或关键数据流 | 架构设计文档、对应开发规范                              |
| 新增、删除或修改 API                     | 对应接口 Markdown、Route Handler 顶部注释、所属功能文档 |
| 修改数据库表、字段或统计口径             | `sql/init.sql`、架构设计文档、所属功能或技术文档        |
| 修改开发约束、工具链或质量门禁           | 对应开发规范、规范导航和根 README                       |
| 实施现有设计方案                         | 对应方案、产品文档和技术文档                            |

只调整内部实现且不改变用户行为或架构边界时，无需机械修改全部文档，但必须确认已有描述仍然准确。

## 8. 文档状态管理

- 当前实现基线：描述已经落地并可验证的系统事实。
- 当前功能说明：描述用户现在可以使用的功能和行为。
- 开发规范：约束代码、文档和协作方式。
- 设计方案：记录尚未实施的设计，不混入当前功能基线。
- 已废弃：内容不再适用；只有存在保留价值时才移动到 `99-归档/`。

没有历史价值的过期文档直接删除，不同时保留多份内容相近的有效文档。
