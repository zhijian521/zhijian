## 架构概览

```
登录/注册 → API 设置 session cookie
              │
              ▼
浏览器 cookie（zhijian_session）
              │
              ├── Server Component → getSessionFromCookies() → requireAdmin()
              ├── API Route → getSessionFromRequest() → withAdmin() / withUser()
              └── Client Component → useAuth() → /api/auth/me
```

两种角色：

| 角色    | 用途           | 登录入口              |
| ------- | -------------- | --------------------- |
| `admin` | 后台管理       | `/admin/login`        |
| `user`  | 导航页数据存储 | `/nav` 内嵌 AuthModal |

## Session Token

**格式**：`userId:username:role:expiresAt.signature`

- 签名算法：HMAC-SHA256，密钥为 `ADMIN_SESSION_SECRET` 环境变量
- 有效期：7 天（`SESSION_MAX_AGE = 604800`）
- 存储：`httpOnly` cookie，名称 `zhijian_session`
- Cookie 配置：`sameSite: 'lax'`，生产环境 `secure: true`

**解析流程**：

1. 按 `.` 分割为 payload 和 signature
2. `safeEqual()` 常量时间比较签名，防时序攻击
3. 解析 payload 四段，校验格式和过期时间
4. 返回 `SessionPayload` 或 `null`

## API 接口

### POST /api/auth/login

公开接口，无需鉴权。

| 字段       | 类型   | 必填 | 说明     |
| ---------- | ------ | ---- | -------- |
| `username` | string | 是   | 用户名   |
| `password` | string | 是   | 明文密码 |

**流程**：查询用户 → 校验状态（disabled 拒绝）→ bcrypt 验证密码 → 生成 token → 设置 cookie → 返回用户信息。

**错误码**：`BAD_REQUEST`（空字段）、`UNAUTHORIZED`（账号或密码错误）、`USER_DISABLED`（账号已禁用）、`RATE_LIMITED`（尝试过于频繁，HTTP 429）。

### POST /api/auth/register

公开接口，无需鉴权。新用户默认角色 `user`。

| 字段       | 类型   | 必填 | 说明                |
| ---------- | ------ | ---- | ------------------- |
| `username` | string | 是   | 2-50 字符，不含 `:` |
| `email`    | string | 是   | 须含 `@`，最长 255  |
| `password` | string | 是   | 至少 6 字符         |

**错误码**：`BAD_REQUEST`（字段校验失败）、`USER_EXISTS`（用户名或邮箱重复，`ER_DUP_ENTRY`）、`INTERNAL`（服务器错误）、`RATE_LIMITED`（尝试过于频繁，HTTP 429）。

### POST /api/auth/logout

清除 session cookie（`maxAge: 0`）。无需鉴权。

### GET /api/auth/me

返回当前登录用户信息。通过 cookie 解析 session，再查数据库确认用户仍有效。

**错误码**：`UNAUTHORIZED`（未登录）、`USER_DISABLED`（用户已禁用）。

## 鉴权守卫

### Server Component 守卫

| 函数                       | 用途                   | 未通过行为                               |
| -------------------------- | ---------------------- | ---------------------------------------- |
| `requireAuth(redirectTo?)` | 要求已登录（任意角色） | 重定向到 `redirectTo` 或 `/admin/login`  |
| `requireAdmin()`           | 要求 admin 角色        | 未登录 → 登录页；非 admin → `/forbidden` |

### API Route 包装器

| 包装器               | 用途                   | 未通过行为    |
| -------------------- | ---------------------- | ------------- |
| `withAdmin(handler)` | 要求 admin 角色        | 返回 403 JSON |
| `withUser(handler)`  | 要求已登录（任意角色） | 返回 401 JSON |

### 会话吊销（validateSession）

token 只证明身份来源可信，所有守卫（`requireAuth` / `requireAdmin` / `withAdmin` / `withUser`）在解析 token 后都会经 `validateSession()` 回查数据库：

- 用户必须存在且 `status = 'active'`，否则视为未登录（禁用/删除立即失效）
- 返回的 `SessionPayload.role` **以数据库为准**，admin 被降级后即刻失去后台权限
- 60 秒内存缓存避免每请求查库；`updateUser` / `deleteUser` 会主动失效缓存，吊销实时生效
- 数据库不可用或查不到用户时不写缓存，避免故障期结果被固化

**用法示例**：

```typescript
// withAdmin: handler 第二个参数为 admin (SessionPayload)
export const GET = withAdmin(async (request, admin, { params }) => {
    // admin.userId, admin.username, admin.role
    return NextResponse.json(success(data));
});

// withUser: handler 第二个参数为 user (SessionPayload)
export const GET = withUser(async (request, user, { params }) => {
    return NextResponse.json(success(data));
});
```

## 客户端 Hook

**文件**：`src/hooks/use-auth.ts`

`useAuth()` 提供登录/注册/登出/状态查询，用于导航页 AuthModal 组件。

```typescript
const { user, isLoggedIn, loading, login, register, logout } = useAuth();
```

| 返回值                                | 类型                                            | 说明               |
| ------------------------------------- | ----------------------------------------------- | ------------------ |
| `user`                                | `AuthUser \| null`                              | 当前用户信息       |
| `isLoggedIn`                          | `boolean`                                       | 是否已登录         |
| `loading`                             | `boolean`                                       | 首次加载中         |
| `login(username, password)`           | `(string, string) => Promise<AuthUser>`         | 登录，失败抛错     |
| `register(username, email, password)` | `(string, string, string) => Promise<AuthUser>` | 注册，失败抛错     |
| `logout()`                            | `() => Promise<void>`                           | 登出，清空本地状态 |

初始化时自动调用 `GET /api/auth/me` 获取登录状态。

## 安全措施

| 措施            | 说明                                    |
| --------------- | --------------------------------------- |
| 密码哈希        | bcrypt 12 rounds，不存明文              |
| Token 签名      | HMAC-SHA256，密钥从环境变量读取         |
| 时序攻击防护    | `crypto.timingSafeEqual` 比较签名       |
| httpOnly cookie | JS 无法读取 session token               |
| SameSite=Lax    | 防 CSRF，允许顶级导航                   |
| 生产环境 Secure | 仅 HTTPS 传输                           |
| 登录错误统一    | 「账号或密码错误」不透露具体原因        |
| 登录/注册限流   | 登录按 IP+用户名、注册按 IP，5 次/分钟，超限返回 429；IP 来源见下文「限流的 IP 来源策略」 |
| 禁用账号拦截    | 登录和 /me 均检查 status                |
| 会话吊销        | 守卫经 `validateSession` 回查库，禁用/降级/删除立即生效（60 秒缓存，变更主动失效） |
| 环境变量必需    | `ADMIN_SESSION_SECRET` 未设置时阻止启动 |

### 限流的 IP 来源策略

限流键中的 IP 由 `src/lib/core/request-ip.ts` 的 `getClientIp(request)` 统一提取：

1. 优先取 `x-real-ip`——生产部署中由 Nginx `proxy_set_header X-Real-IP $remote_addr` 覆写，客户端自带的同名头不会透传，无法伪造；
2. 否则取 `x-forwarded-for` 链的**最右侧**非空值——单跳可信反代会把真实 IP 追加在链尾，而链首由客户端原始请求携带、可任意伪造，不可取；
3. 两者皆无或格式不合法时返回 `'unknown'`。

**部署假设**：Nginx 单跳反代 + `next start`（见 [07-部署指南](07-部署指南.md)）。若变为多跳反代或客户端直连 Node，需重新评估取值策略。

**降级行为**：登录接口在 IP 为 `'unknown'` 时降级为仅按用户名（截断前 50 字符）限流，并 `console.warn` 记录；注册接口以 `'unknown'` 作为共享限流键（宁可误伤也不放行批量注册）。
