# 01 · 项目总则

- 适用范围：当前项目
- 最后核对：2026-07-18
- 维护范围：项目目标、技术栈、边界和统一完成标准
- 关联代码：`package.json`、`src/`、`scripts/`

## 目标

「知简」是一个个人博客 + 导航站 + 后台管理的 Next.js 站点。本项目以清晰结构、稳定边界和可执行规范作为长期维护基础。

核心目标只有三个：

- 简洁：能复用就不重写，能删掉就不保留，能一层解决就不绕两层。
- 整齐：同类东西放在同类位置，命名、分层、依赖方向稳定一致。
- 完整：功能闭环，接口有文档，组件可预览，规则有校验兜底。

## 技术栈

| 项       | 选型                                                 | 备注                             |
| -------- | ---------------------------------------------------- | -------------------------------- |
| 框架     | Next.js 15 App Router                                | 不使用 Pages Router              |
| 语言     | TypeScript 5                                         | 严格模式                         |
| 运行时   | Node.js                                              | API Route 跑在服务端             |
| 数据库   | MySQL + `mysql2/promise`                             | 连接池单例                       |
| 样式     | CSS Modules + 原生 CSS 变量                          | 不引入 Tailwind / CSS-in-JS      |
| Markdown | `react-markdown` + `remark-gfm` + `rehype-highlight` | 博客正文、导航笔记和 AI 内容共用 |
| 图表     | `recharts`                                           | 仅后台分析使用                   |

新增依赖前，先按这个顺序自问：

1. 现有实现能不能复用？
2. Node/浏览器原生能力能不能解决？
3. 已安装依赖能不能解决？
4. 如果都不行，再引新依赖，而且必须能说明收益。

## YAGNI 优先级

写任何东西前，按这个顺序判断，停在第一个成立的位置：

1. 这东西现在真的需要吗？
2. 项目里已经有了吗？
3. 标准库能做吗？
4. 平台原生能做吗？
5. 已装依赖能做吗？
6. 能不能用最小实现做完？

不要为了“以后可能用到”提前建接口、目录、抽象层、工厂或空文档。

## 项目边界

- 项目以博客、导航、后台三块主要功能域组织，公共能力按统一目录规范沉淀。
- 调整实现时优先保持产品行为稳定，不为了“更优雅”随意改动既有交互、接口语义或数据规则。
- 目录结构、模块边界、规范体系和代码组织方式必须长期稳定，避免随功能推进持续漂移。
- 如果需要调整架构边界，必须同步更新相关开发规范、技术文档和校验脚本。

## 明确不做

- 不做全自动反射式 showcase / API 扫描。
- 不做全局 `modules/` 空壳 CSS。
- 不做只有一个实现的接口抽象。
- 不做没有落地校验的“建议型规范”。

## 强制校验

提交前统一运行：

```bash
npm run check
```

当前 `check` 至少包含：

- `npm run lint`
- `npm run typecheck`
- `npm run docs:check`

如果只是大范围整理格式，可额外运行：

```bash
npm run format
```

但“这次改动是否完成”的判断，以 `npm run check` 为准。
