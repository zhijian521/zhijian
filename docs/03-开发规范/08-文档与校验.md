## 文档分类

| 类型     | 位置                  | 职责                                             |
| -------- | --------------------- | ------------------------------------------------ |
| 文档导航 | `docs/00-文档导航.md` | 全部文档入口、阅读顺序和维护规则                 |
| 产品文档 | `docs/01-产品文档/`   | 当前需求、功能地图和用户可见行为                 |
| 技术文档 | `docs/02-技术文档/`   | 架构、认证、接口、导出、同步和部署               |
| 开发规范 | `docs/03-开发规范/`   | 目录、代码、样式、接口、数据和组件规则           |
| 方案文档 | `docs/04-方案文档/`   | 尚未实施但需要长期保留的设计，仅在真实需要时创建 |

根目录 `README.md` 是项目入口，根目录 `AGENTS.md` 是 AI 与开发工具速查。所有正式文档统一放在 `docs/`，不参与 Next.js 运行时，也不在后台单独渲染。

## 当前实现与方案分离

- 需求、功能、架构和专项文档只描述已经落地、可以从代码验证的事实。
- 尚未实现的设计放入方案文档，并明确标记“设计方案”。
- 方案落地后，把最终行为同步到产品和技术文档；没有历史价值的旧方案直接删除。
- 不允许把未来计划提前写入“当前实现基线”。

## API 文档

API 文档由两部分组成：

1. `docs/02-技术文档/03-接口文档/` 下的 Markdown。
2. 对应 `src/app/api/**/route.ts` 顶部注释。

Markdown 提供按领域阅读的接口索引、鉴权、方法、参数和响应；Route Handler 注释把关键信息保留在实现旁边。新增或修改接口时二者必须同步。

接口 Markdown 的二级标题使用固定格式：

```markdown
## /api/admin/posts · 文章管理（列表/创建）

### GET · 分页获取文章

### POST · 创建草稿
```

二级标题登记路径，三级标题按 `### HTTP方法 · 说明` 登记方法。不同方法的参数、状态码和响应必须分别说明，避免共享描述掩盖契约差异。

`docs:check` 通过上述标题提取路径和 HTTP 方法，与磁盘上的 Route Handler 双向核对。

## docs:check 负责什么

`scripts/docs-check.mjs` + `npm run docs:check` 当前负责：

1. 文档导航、需求、功能、架构、规范导航和接口导航存在。
2. `docs/**/*.md` 中的相对 Markdown 链接指向真实文件。
3. 接口 Markdown 中登记的每个 `/api/...` 都存在对应 Route Handler。
4. 磁盘上的每个 `src/app/api/**/route.ts` 都登记在接口 Markdown 中。
5. 接口 Markdown 登记的 HTTP 方法与 Route Handler 实际导出的方法双向一致。
6. 不允许同一路径或同一路径下的同一方法重复登记。

校验保持轻量，不自动判断鉴权、参数和响应语义，也不从实现生成产品描述；这些内容仍需在代码审查中人工核对。

## 代码变更同步矩阵

| 代码变化                         | 同一提交内必须检查并更新                                |
| -------------------------------- | ------------------------------------------------------- |
| 新增或删除产品功能               | 需求文档、功能文档、对应功能模块文档                    |
| 页面、路由、交互或权限变化       | 功能文档、对应功能模块文档                              |
| 分层、目录、依赖或关键数据流变化 | 架构设计文档、对应开发规范                              |
| API 新增、删除或契约变化         | Route Handler 顶部注释、对应接口 Markdown、所属功能文档 |
| 数据表、字段或统计口径变化       | `sql/init.sql`、架构设计文档、所属功能或技术文档        |
| 校验脚本或统一完成标准变化       | 本规范、规范导航、文档导航和根 README                   |

内部重命名、无行为影响的样式调整或局部实现优化不要求机械修改全部文档，只更新真实受影响的内容。

## 提交流程

提交前统一运行：

```bash
npm run check
```

它是当前统一完成标准。单独运行 `npm run docs:check` 只适合文档迁移或接口登记时快速验证，不替代最终检查。
