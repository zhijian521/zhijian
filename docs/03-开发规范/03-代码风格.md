# 03 · 代码风格

- 文档状态：开发规范
- 适用范围：当前项目的 Next.js、React、TypeScript 代码
- 最后核对：2026-07-21
- 维护范围：文件结构、导入、空行、控制流、命名、注释、TypeScript 和工具链规则
- 关联代码：`src/`、`.editorconfig`、`.eslintrc.json`、`.prettierrc.json`

本文用于统一项目编码习惯，提高可读性并降低维护成本。仓库中的 ESLint、TypeScript 和 Prettier 配置是可执行基线；本文负责补充工具无法自动判断的结构和表达规则。

## 核心原则

代码遵循四个原则：

1. **统一**：同一种代码在整个项目中保持同一种写法，一致性优先于个人偏好。
2. **清晰**：阅读代码的速度比编写代码的速度更重要。
3. **简单**：优先选择最容易理解、依赖最少、嵌套最浅的实现。
4. **可维护**：命名和结构负责表达“做什么”，注释只补充“为什么”。

开始实现前先查找已有组件、Hook、数据函数和工具。不要为同一职责引入平行实现，也不要为尚不存在的需求预留空抽象。

## 文件结构

页面和较长模块按以下顺序组织；没有对应内容时直接省略，不创建空 Section：

```text
文件头
↓
Imports
↓
Metadata
↓
Config / Constants
↓
Page / Component
↓
Helpers
↓
Types（无法就近表达时）
```

页面示例：

```tsx
/*============================================================================
  Admin Login Page

  已登录：
  - admin -> /admin
  - user -> /

  未登录：
  - 渲染登录页面
============================================================================*/

import type { Metadata } from 'next';

import { redirect } from 'next/navigation';

import AdminLoginCard from '@/components/modules/admin/admin-login-card';

import { getSessionFromCookies, validateSession } from '@/lib/core/auth';
import { APP_ROUTES } from '@/lib/core/site';

// =============================================================================
// Metadata
// =============================================================================

export const metadata: Metadata = {
    title: '后台登录',
};

// =============================================================================
// Page
// =============================================================================

export default async function AdminLoginPage() {
    const session = await validateSession(await getSessionFromCookies());

    if (!session) {
        return <AdminLoginCard />;
    }

    if (session.role === 'admin') {
        redirect(APP_ROUTES.admin);
    }

    redirect(APP_ROUTES.home);
}
```

规则：

- 文件头说明文件职责、角色分流、数据边界或其他关键约束，不复述文件名。
- 简短且职责单一的文件不强制添加文件头或 Section。
- 较长文件在存在多个独立逻辑区时使用 Section，不能只按行数机械切段。
- 类型优先靠近使用处；跨路由共享类型才放入 `src/types/`。

## Import 规范

Import 按以下顺序分组，组与组之间保留一个空行：

1. 类型导入：使用 `import type`。
2. Node.js、React 和 Next.js。
3. 第三方库。
4. Components：`@/components/*`。
5. Hooks：`@/hooks/*`。
6. 领域服务和数据访问：`@/lib/domain/*`。
7. 核心工具和基础设施：`@/lib/core/*`。
8. Constants / Config，以及同目录相对导入和 CSS Module。

```tsx
import type { Metadata } from 'next';

import { redirect } from 'next/navigation';

import clsx from 'clsx';

import AdminLoginCard from '@/components/modules/admin/admin-login-card';

import { useAuth } from '@/hooks/use-auth';

import { getPostById } from '@/lib/domain/posts';

import { validateSession } from '@/lib/core/auth';
import { APP_ROUTES } from '@/lib/core/site';

import styles from './page.module.css';
```

补充规则：

- 同组导入保持职责相近，避免仅为字母排序打散有意义的组合。
- 跨目录统一使用 `@/*`，同目录文件使用 `./`；不要编写多层 `../../` 路径。
- Import 本身已经能表达来源时，不额外添加“组件导入”“数据导入”等注释。
- 只有实际存在的组才保留空行，不为缺失分组创建占位注释。

## 空行规范

空行表示逻辑块，不用于装饰。

推荐：

```tsx
const session = await validateSession(await getSessionFromCookies());

if (!session) {
    return <AdminLoginCard />;
}

const user = await getUserById(session.userId);

return <Profile user={user} />;
```

相邻语句属于同一处理步骤时保持紧凑，不要在连续声明之间随意插入空行。`return` 前保留一个空行，除非函数只有一条直接返回语句。

## 函数与对象

函数保持职责单一，参数或调用超过行宽、包含多个复杂表达式或无法快速扫描时拆行。短调用保持单行，最终格式以 Prettier 为准，不手工制造与格式化工具反复冲突的换行。

```tsx
const result = await updateUser(userId, {
    role: nextRole,
    status: nextStatus,
    password: nextPassword,
});
```

对象属性一行一个，保留尾逗号；极短的空对象或由 Prettier 保留的单行对象除外。

```tsx
const config = {
    title: '后台',
    cache: false,
    redirect: true,
    timeout: 3000,
};
```

## 控制流

优先使用 Guard Clause 和 Early Return 处理失败、空值、权限与边界条件，减少嵌套。

推荐：

```tsx
if (!session) {
    return <AdminLoginCard />;
}

if (session.role === 'admin') {
    redirect(APP_ROUTES.admin);
}

redirect(APP_ROUTES.home);
```

不推荐：

```tsx
if (session) {
    if (session.role === 'admin') {
        redirect(APP_ROUTES.admin);
    }

    redirect(APP_ROUTES.home);
}

return <AdminLoginCard />;
```

补充规则：

- 不嵌套三元运算符；多分支使用 Guard Clause、`if` / `else`、`switch` 或拆分函数。
- 单一条件且表达式简短时可以使用三元运算符。
- 异步流程需要并发时显式使用 `Promise.all`；存在先后依赖时保持顺序执行。

## 注释与 Section

注释解释意图、业务规则、异常约束或技术取舍，不翻译代码表面行为。

不推荐：

```tsx
// 获取 Session
const session = await getSession();
```

推荐：

```tsx
// 回查数据库，避免已禁用或删除的账号继续沿用旧 Session
const session = await validateSession(await getSessionFromCookies());
```

项目使用以下注释层级：

| 形态                  | 用途                                   |
| --------------------- | -------------------------------------- |
| `/*===== ... =====*/` | 文件头，说明职责和关键约束             |
| `// =====` 三行格式   | 较长文件的主要 Section                 |
| `/*== xxx ==*/`       | 主要 Section 内确有必要的紧凑子区      |
| `{/* xxx */}`         | JSX 中不易从结构看出的语义区块         |
| `/** ... */`          | 需要 IDE 悬浮说明的公共 API 或复杂参数 |

主要 Section 使用统一格式：

```tsx
// =============================================================================
// Helpers
// =============================================================================
```

规则：

- 文件较长且包含多个职责时再添加 Section，通常包括 `Metadata`、`Config`、`Page`、`Helpers`、`Types` 或 `Render`。
- 不添加 `// if 判断`、`// return`、`// 组件导入` 等可由代码直接读出的注释。
- 注释必须随逻辑同步更新；过期注释比没有注释更危险。
- 不用注释保留死代码，直接删除并交由 Git 记录历史。

## 命名规范

| 对象              | 规则                                 | 示例                               |
| ----------------- | ------------------------------------ | ---------------------------------- |
| 文件 / 目录       | kebab-case                           | `api-response.ts`、`ghost-button/` |
| 组件文件          | kebab-case；导出名 PascalCase        | `ghost-button.tsx`、`GhostButton`  |
| 函数 / 变量       | camelCase，名称表达具体职责          | `validateSession`、`currentUser`   |
| Boolean           | 使用 `is` / `has` / `can` / `should` | `isLoading`、`hasPermission`       |
| 类型 / 接口       | PascalCase                           | `SessionPayload`                   |
| 常量对象          | PascalCase                           | `BizCode`                          |
| CSS 变量          | `--kebab-case`                       | `--space-4`                        |
| CSS Modules class | camelCase                            | `.submitButton`                    |

函数名优先使用明确动词，例如 `getSession`、`validateSession`、`createUser`、`deletePost`。避免 `handleData`、`processInfo`、`isLogin` 等含义模糊或语法不完整的名称；状态应写成 `isLoggedIn`。

## JSX 规范

无 Props 或只有短小 Props、且 Prettier 能稳定保留时使用单行：

```tsx
<AdminLoginCard />
```

多个 Props 或内容较长时每行一个 Props：

```tsx
<LoginCard description="请输入账号密码" loading={isLoading} title="后台登录" />
```

补充规则：

- Props 使用语义名称，Boolean Props 优先直接写 `disabled`，不写 `disabled={true}`。
- 列表渲染使用稳定业务标识作为 `key`，不能在可重排列表中使用数组索引。
- 条件渲染复杂时先计算具名变量或拆组件，不在 JSX 中堆叠多层表达式。
- 页面负责路由和数据编排，复杂交互移入 `components/modules/`，共享展示移入 `components/site/` 或 `components/ui/`。

## TypeScript

- 开启严格模式，公共函数显式标注返回类型。
- 默认禁止 `any`；不可信输入先使用 `unknown`，在边界完成收窄和校验。
- 优先使用类型推断，不重复标注显而易见的局部变量类型。
- 类型尽量就近定义，不创建大杂烩类型文件。
- 使用 `import type` 隔离纯类型依赖。
- 不用非空断言掩盖缺失状态；先证明值存在或提供明确回退。

## 工具链

| 命令                   | 用途                         |
| ---------------------- | ---------------------------- |
| `npm run format`       | Prettier 直接格式化          |
| `npm run format:check` | 检查格式是否一致             |
| `npm run lint`         | ESLint 校验                  |
| `npm run typecheck`    | TypeScript 校验              |
| `npm run docs:check`   | Markdown 链接与 API 文档校验 |
| `npm run check`        | 统一完成前校验入口           |

默认格式：

- 4 空格缩进。
- 单引号和分号。
- 对象、数组等多行结构保留尾逗号。
- 最大行宽 120；长行和稳定换行交给 Prettier。
- 不手工用空格对齐标识符、参数或对象值。

## 简洁与维护

- 先复用现成实现，再考虑新增代码。
- 重复的鉴权、错误响应或数据编排优先使用项目已有 wrapper / helper。
- 函数保持单一职责；当多个步骤可以独立命名和验证时再拆分。
- 不做只有一个实现、没有降低复杂度的抽象。
- 不使用空接口、空实现、空目录或未来占位模块。
- 修改旧文件时遵循本规范，但不为统一格式顺带重写无关模块。

## 最小自检

非平凡逻辑需要留下与当前仓库能力匹配的可执行验证：

- 优先运行或补强已有的 lint、typecheck、`docs:check` 和构建校验。
- 纯函数、解析、权限、安全或复杂状态逻辑应补充可重复运行的自动化测试。
- UI 行为按风险使用浏览器验证，检查桌面端、移动端、控制台和关键交互。
- 在没有测试框架时，不创建没人会执行的孤立测试文件；应同时补充可靠的执行入口。

## 禁止项

- 禁止提交调试用 `console.log`。
- 禁止没有上下文的 `TODO`；确需保留时写明负责人或范围与原因。
- 禁止注释显而易见的 `if`、赋值、循环和 `return`。
- 禁止通过深层嵌套表达可用 Guard Clause 展开的流程。
- 禁止为了“看起来整齐”添加无逻辑意义的空行或手工对齐。
- 禁止新增与 ESLint、TypeScript、Prettier 或项目分层相冲突的个人写法。

## 提交前检查

- Import 是否按固定职责分组。
- 空行是否只用于分隔逻辑块。
- 控制流是否可以用 Guard Clause 降低嵌套。
- 注释是否解释 Why，而不是重复 What。
- 文件是否遵循“Header → Imports → Metadata / Config → Page / Component → Helpers → Types”。
- 命名是否准确，Boolean 是否使用 `is` / `has` / `can` / `should`。
- 是否已运行 `npm run check`，并按变更风险补充针对性验证。
